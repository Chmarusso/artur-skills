# Make PR

Push the current branch and create a pull request, or update an existing one. PR descriptions focus on user-facing value — what problems are solved and what benefits users get.

## Instructions

1. **Get current branch and check for existing PR:**
   - Run `git branch --show-current` to get the branch name.
   - If on `main` or `master`, tell the user and stop.
   - Run `gh pr view --json number,url,state 2>&1` to check if a PR already exists for this branch.

2. **Commit uncommitted changes (if any):**
   - Run `git status --short` to check for staged and unstaged changes.
   - If the working tree is clean, skip to the next step.
   - If there are uncommitted changes, split them into small, isolated commits:
     - Group related changes by feature, fix, or file area — each commit should be one logical unit.
     - For each group: stage only those files (`git add <files>`), then commit with a single-line summary message.
     - Commit messages should be lowercase, imperative, concise (e.g., `update redirect URLs to unified login`, `remove unused creator login form`).
     - Prefer several small commits over one large commit. If in doubt, split.
     - Do NOT use `git add -A` or `git add .` — always add specific files.

3. **If a PR already exists:**
   - Push changes: `git push`.
   - Print the existing PR URL and stop. Do NOT update the PR title or description.

4. **If no PR exists, gather context:**
   - Run `git log main..HEAD --oneline` (or `master..HEAD`) to see all commits on this branch.
   - Run `git diff main...HEAD --stat` to see which files changed.
   - Run `git diff main...HEAD` to see the full diff. If the diff is very large, focus on the most important changes.

5. **Push the branch:**
   - Run `git push -u origin HEAD`.

6. **Create the PR:**
   - Write a short title (under 70 characters) that summarizes the change from a user's perspective.
   - Write a description body with this structure:

   ```
   ## Summary
   <2-4 bullet points focusing on what users/stakeholders gain — problems solved, capabilities added, friction removed. Avoid implementation details unless they matter to the user.>

   ## Changes
   <Brief list of the key technical changes, 2-5 bullets. Keep it scannable.>
   ```

   - Create the PR: `gh pr create --title "..." --body "$(cat <<'EOF' ... EOF)"`.
   - Print the new PR URL.

## Writing the description

- Lead with **why this matters**, not what was done. "Users can now sign in from a single page" beats "Merged login forms into one component".
- Mention the problem being solved if it's not obvious from the title.
- Skip boilerplate like "This PR..." — just state the facts.
- Do NOT add "Generated by" footers, test plan sections, or checklists.
- Keep the whole body under 15 lines.
